The user reports that even after chatting, the score isn't showing up in the "Candidate Pool" (Square) for users they've already chatted with.

**Analysis:**
1.  **Issue**: The leaderboard works (shows history), but the **square** (candidate list) is supposed to show "✅ 已聊过 (XX分)" for people you've chatted with. The user implies this isn't happening for new chats.
2.  **Root Cause**:
    *   In `app.py`, `chatted_map = storage.get_chatted_users(current_user.user_id)` is called **before** the sidebar and candidate pool rendering.
    *   `storage.get_chatted_users` queries the `match_records` table.
    *   The query in `get_chatted_users` (lines 186-188) looks correct: `SELECT user_a, user_b, match_score FROM match_records WHERE user_a = :u OR user_b = :u`.
    *   **However**, for *virtual users* (IDs starting with `guest_`), they are NOT in the `users` table.
    *   When we save a match record for a virtual user, we save their ID (e.g., `guest_12345`).
    *   But `get_match_history` (used for the sidebar) does a `JOIN users u` (line 168). **This JOIN will fail for virtual users** because they don't exist in the `users` table!
    *   So, while the record is in `match_records`, `get_match_history` returns nothing for virtual users, and thus they don't appear in the sidebar history.
    *   Also, `get_chatted_users` (used for the square) does *not* do a JOIN, so it *should* return the scores. If the user says "not added to leaderboard", they likely mean the sidebar leaderboard (which uses `get_match_history`).

**Fix for Leaderboard (Sidebar):**
The `get_match_history` function uses an `INNER JOIN` (implied or explicit) with the `users` table to get the partner's name.
*   `JOIN users u ON ...`
*   Since virtual users aren't in `users`, this row is filtered out.

1.  Modify `get_match_history` in `src/storage.py` to use a `LEFT JOIN`.

**Refined Plan for `src/storage.py`**:
Change the SQL in `get_match_history` to:
```sql
SELECT r.*, COALESCE(u.name, 'AI Guest') as partner_name 
FROM match_records r
LEFT JOIN users u ON (u.username = r.user_b AND r.user_a = :u) OR (u.username = r.user_a AND r.user_b = :u)
WHERE r.user_a = :u OR r.user_b = :u
ORDER BY r.match_score DESC
```
This ensures virtual users (who fail the join) still show up in the history.
